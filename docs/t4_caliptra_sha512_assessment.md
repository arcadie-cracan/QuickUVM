# T4 — Caliptra SHA512 (UVMF) vs QuickUVM — generator vs generator

Caliptra is a datacenter silicon root-of-trust (OCP/CHIPS-Alliance; AMD, Google, Microsoft,
NVIDIA). Its SHA512 UVM testbench was **generated by Siemens UVMF** — a tool architecturally
identical to QuickUVM: Python + Jinja2 + YAML + a schema validator + regenerate-over-preserved
`// pragma … custom` regions. Same tool class, on taped-out silicon. This is the only
apples-to-apples generator-vs-generator comparison in the campaign.

Every claim is tagged **[C]** verified by construction (generated and/or elaborated on Xcelium this
session) / **[I]** inspection of a real file / **[P]** predicted, not run. After the T2 review found
nine false claims in a paper write-up that "read fine," nothing here is asserted from a summary.

---

## 1. The boundary statement

**QuickUVM reproduces the entire HVL half of the UVMF SHA512 bench; it structurally cannot
reproduce the HDL/BFM half — and that half is emulation apparatus, a declared non-goal, so the
majority of the gap is gap-by-design, not weakness.**

- **[C] The HVL half is reproducible.** A QuickUVM YAML mirroring the UVMF source (two agents —
  an AHB-driving initiator carrying `op`+`test_case_sel`, and a passive monitor-split carrying the
  512-bit `result` — a golden-model predictor, scoreboard, coverage, the typed `op` enum + its
  constraint) **generates clean (32 files)** and **elaborates on Xcelium (0 errors)**. See
  [`t4_sha512_probe/`](t4_sha512_probe/).
- **[C] The HDL/BFM half is structurally absent.** The real UVMF `SHA512_in_driver_bfm.sv` is a
  synthesizable `interface` module carrying `pragma attribute … partition_interface_xif`,
  `pragma tbx xtf`, `pragma tbx oneway proxy.…`, and `` `ifndef XRTL `` — the Siemens Veloce
  hardware-emulation partitioning apparatus, driving DUT pins tri-state. A `models.py` absence audit
  returns **zero** fields for `bfm`, `proxy`, `xrtl`, `tbx`, `partition` (the lone `partition` hit is
  an unrelated coverage message). QuickUVM has no BFM-in-HDL split and no emulation partitioning, by
  design.

A skeptic accepts this because every "reproduces" clause points at a file generated and elaborated
this session, and every "cannot" clause points at a real UVMF emulation construct **plus** the
absence of a corresponding `models.py` field — not at a QuickUVM failure to try.

## 2. The predicted-inexpressible list — mostly REFUTED

The campaign predicted a set of things QuickUVM's schema could not express. Checked against the real
files and by generating, **most were wrong** — and a wrong prediction is worth as much as a right
one (the campaign's own rule; my "K0 breaks" and "CPOL=1 misaligns" predictions were also wrong).

| predicted-inexpressible | verdict | evidence |
|---|---|---|
| `hdl_typedefs` (the typed `op` enum) | **REFUTED [C]** | generated `sha_in_item.svh`: `typedef enum logic [2:0] {reset_op=0, sha224_op=4 … } op_e;` + `op inside {…}` constraint. Only the *synthesizable HDL copy* of it is inexpressible — because there is no HDL side. |
| explicit `tlm_connections:` (QuickUVM infers) | **REFUTED [C]** | one `{source: sha_in, monitor: sha_out}` line generates exactly UVMF's three edges: `in.ap→pred`, `pred→cmp.expected`, `out.ap→cmp.actual`. Inference is enough for this standard single-predictor shape. |
| an ACTIVE RESPONDER role | **REFUTED [I]** | the real `SHA512_out` interface signals are all **inputs** — a passive monitor-split, not an active responder. Maps to one passive QuickUVM agent. (And the responder role itself shipped in PR #52/#53 anyway.) |
| predictor `$system("python …")` + `$fscanf` | **REFUTED — premise factually wrong [C]** | the real `SHA512_predictor.svh` uses `$fopen`/`$fgets`/`$sscanf` on static NIST `.rsp` vectors. No `$system`, no Python, no DPI. See §3. |
| driver-proxy / HDL-BFM split | **CONFIRMED [I]** | UVMF's HVL driver holds a `virtual …_driver_bfm bfm` + struct marshaling; QuickUVM's driver is one plain class on the vif. Emulation-only. |
| `partition_interface_xif` emulation partitioning | **CONFIRMED [C]** | real BFM attributes (§1) vs zero `models.py` knobs. Emulation-only, by design. |

**Net: two confirmed inexpressible (both emulation-only, by design); four refuted.** The refutations
are the most useful content here — the campaign over-predicted QuickUVM's schema gaps.

## 3. The predictor is a QuickUVM win — and *easier* than HMAC

The campaign's "bonus" premise was that Caliptra's predictor shells out to Python. **It does not.**
The real predictor reads NIST CAVP `.rsp` files with plain SV I/O:

    fd_r = $fopen(file_name,"r");
    while (cnt < line_skip) begin cnt++; $fgets(line_read,fd_r); end
    $sscanf(line_read, "%s %s %h", tmp1, tmp2, expected);
    result = expected << shift;

**[C] Ported into QuickUVM's K0 seam, it elaborates on Xcelium and survives idempotent
regeneration.** The mode-select + file-read + shift live in the `prediction_logic` pragma; the
handles/scratch in `class_item_additional`. Regenerating is a no-op (`0 updated, 32 unchanged`,
byte-identical diff), and the filled bench elaborates with **0 errors**. Because the mechanism is
native SV file I/O — no DPI, no >64-bit marshaling, no `chandle` — T1's known limit (the `language:
c` DPI bridge is scalar-only) **does not bite**: SHA512's predictor is *strictly easier* for
QuickUVM than HMAC's was.

## 4. The metric — deliberately demoted

Both benches are machine-generated, so a hand-written-line ratio is tempting. It is also, here,
**too caveated to be a headline** — and presenting it as "QuickUVM is leaner" is exactly the
flattering-but-misleading move the T2 review caught. What I can stand behind, one rule (non-blank
lines strictly inside pragma regions ÷ total generated lines):

| | hand-in-pragma | total | ratio | provenance |
|---|---:|---:|---:|---|
| QuickUVM SHA512 | 69 | 1,425 | 4.8% | **[C]** measured this session |
| QuickUVM HMAC (T1) | 159 | 1,431 | 11.1% | **[C]** measured |
| UVMF SHA512 | 1,087 | 6,238 | 17.4% | **[I]** *their* reported figure — I did **not** recount 56 files |

**Why this table does not support a "leaner" claim, stated plainly:**
- **My SHA512 number is a lower bound.** The predictor is filled; the AHB driver body is stubbed.
  A complete bench would hand-write the driver protocol too, raising 69. So 4.8% is not the true
  hand surface.
- **I did not recount UVMF's 56 files.** The 17.4% is their figure; I verified the predictor and
  interface against real source, not the aggregate count.
- **~69% of UVMF's hand-lines are the emulation half QuickUVM omits by design** (their own
  breakdown), so the totals are not like-for-like at all.
- **A lower ratio is not "better verification."** All verification *value* is in the human predictor
  and driver bodies, comparable and hand-written on both sides. A lower ratio means the generator
  carries more scaffolding — an ergonomics result, not a bug-finding one — and a verbose template
  lowers its own ratio for free.

The honest one-liner: **both tools reduce a UVM bench to a small hand-written core plus a generated
shell; on the HVL half QuickUVM's generated shell carries somewhat more of the structure. That is an
ownership/ergonomics observation, not a verification-quality one, and the exact ratio is not
reliable enough to rank the tools.**

## 5. Honest limits

- **No SHA512 DUT, no NIST vectors committed.** The bench elaborates; it is not run to a data-path
  pass (there is no SHA512 RTL here and the comparison is architectural — the campaign says so). The
  predictor claim is "the seam expresses and compiles the file-reading golden model," proven by
  elaboration + idempotent regen — **not** "it computes correct digests against a DUT."
- **UVMF's aggregate metric is not independently recounted** (§4).
- **The QuickUVM metric is predictor-only** (driver stubbed), a lower bound (§4).
- **Not built to a full bench.** This is a construction-grounded finding, like T3 — the value is the
  boundary statement and the refutations, both fully evidenced; the ~8–12 day full build was judged
  over-investment against a majority-by-design gap.
