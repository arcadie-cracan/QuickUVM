# alu — combinational ALU example (typed enum fields; S1 + black box)

A combinational ALU (`W=8`): 8 ops (ADD, SUB, AND, OR, XOR, SLL, SRL, SLT) with
Z/C/N/V flags. Three inputs (`a`, `b`, `op`), five outputs (`result` + 4 flags).
Exercises `dut.combinational: true` on a richer shape with non-trivial flag logic,
and is the worked example for **S1** (typed/enum transaction fields).

## Named opcodes on both sides — but as *separate* types (black box)
Operations are **named** rather than magic numbers (coding guideline: prefer
enums/named constants — `docs/style_systemverilog.md`). Crucially, the DUT and the
testbench name them with **independent** types:

- the **DUT** uses its own `alu_pkg::opcode_e` (its private RTL package);
- the **testbench** declares its **own** `op_e`, generated by QuickUVM from the
  `enum:` block on the `op` port in `alu.yaml` — the TB never imports `alu_pkg`.

This is the **black-box** discipline: the TB encodes the interface spec (opcode
`0=ADD … 7=SLT`) on its own, so a wrong DUT encoding is *caught*, not mirrored. If
the TB shared the DUT's type, the golden model would inherit the same bug and the
test would falsely pass. Proof: swap the DUT's `SRL`/`SLT` encodings in
`rtl/alu_pkg.sv` and re-run — the scoreboard reports ~216 failures.

## S1 in action
The `op` field is now a `rand op_e op` (typed enum), so it **self-constrains** to
the 8 legal opcodes — no hand-written `op inside {…}` constraint in the sequence —
and the golden model reads `case (t.op) ADD: …` with no DUT import.

## Layout
- `rtl/alu_pkg.sv` — the DUT's private `opcode_e` enum (RTL only).
- `rtl/alu.sv` — clean MIT parameterized DUT (`case (opcode_e'(op)) ADD: …`).
- `alu.yaml` — config (`combinational: true`; `op` has an `enum:` → TB-owned `op_e`).
- `gen/` — generated TB; user code:
  - `alu_seq_item.svh` — QuickUVM-generated `op_e` typedef + `rand op_e op`.
  - `alu_tb_pkg.sv` `imports` pragma — empty (no DUT package; the TB owns `op_e`).
  - `alu_reference_model.svh` `prediction_logic` — golden `predict()` model, the spec by `op_e` names.
  - `alu_seq.svh` `do_item_constraints` — empty (the enum self-constrains).
- `sim/xrun.f` — Xcelium filelist (`alu_pkg.sv` compiled before `alu_tb_pkg`, for the DUT).

## Functional coverage (V1)
`alu.yaml` also carries a `coverage_models:` block, so `gen/alu_cov.svh` has a
real covergroup instead of the generic stub: `op` auto-bins one-per-opcode, `a`/`b`
get corner bins (`zero`/`max`/`mid`), `carry` covers `{0,1}`, and `op × a` / `op × b`
crosses ask whether each opcode was exercised against each operand corner. Same
black-box discipline as the opcodes — the bins encode the *spec's* interesting
values, not DUT internals.

```bash
cd sim && xrun -f xrun.f +UVM_TESTNAME=rand_test -coverage functional
# -> TEST PASSED, 1001/1001; covergroup `cov` ≈ 55.56% (28/64 bins) under pure random
```

## Run

