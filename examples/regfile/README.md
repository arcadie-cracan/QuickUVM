# regfile — runnable RAL bench + the CSR test library (C5 + C4)

A tiny register block behind a minimal single-cycle bus, wired to a UVM register
model (RAL) and exercised by the **C5 CSR test library**. This is QuickUVM's first
*runnable* RAL example: it fills the long-standing "no end-to-end register bench"
gap and is the worked example for **C5** (and C4a/b/c — adapter, predictor,
backdoor).

## The DUT and the bus
`rtl/regfile.sv` is four 16-bit R/W registers at byte offsets 0/2/4/6 with distinct
reset values (`0x0000`, `0x00FF`, `0xDEAD`, `0xBEEF`, so the `hw_reset` test has
something to check). One access per cycle on a minimal bus: `addr`/`wdata`/`wr` →
`rdata` (combinational read). The `host` agent in `regfile.yaml` drives it.

## The register model (external, reggen-style)
`ral/regfile_ral_pkg.sv` is a hand-written `uvm_reg_block` — in a real flow this is
generated by reggen / PeakRDL / SystemRDL; here it is hand-written to keep the
example self-contained. QuickUVM consumes it **by name** (`register_model.package` /
`.block`), builds/locks it in the base test, and wires the generated adapter +
explicit-prediction predictor. The four registers carry **backdoor HDL slices**
(`regs[0..3]`); `register_model.backdoor_root: tb_top.dut_inst` makes the model's
auto (DPI) backdoor resolve to the DUT instance.

## C5 — the CSR test library
`csr_tests: [hw_reset, bit_bash, rw]` generates one `regfile_csr_<kind>_test` per
kind, each running the matching UVM built-in register sequence on the locked RAL
with the data-path scoreboard disabled (the RAL is the checker). Run via
`+UVM_TESTNAME`:

| Test | UVM sequence | What it checks |
|---|---|---|
| `regfile_csr_hw_reset_test` | `uvm_reg_hw_reset_seq`   | every register reads its reset value after reset (frontdoor) |
| `regfile_csr_bit_bash_test` | `uvm_reg_bit_bash_seq`   | each R/W bit is independently writable/readable (frontdoor) |
| `regfile_csr_rw_test`       | `uvm_reg_access_seq`     | frontdoor writes agree with **backdoor** reads, and vice-versa |

`rw` (`uvm_reg_access_seq`) is the reason the backdoor matters: without an HDL path
it would warn "no backdoor mechanism" and silently skip every register. With
`backdoor_root` set, it peeks/pokes `regs[]` directly and compares against the
frontdoor.

## A real data-path scoreboard too (`rand_test`)
The default `rand_test` is *not* vacuous: `gen/regfile_reference_model.svh` carries a
golden model of the regfile (four registers, read-then-write per cycle), so the
data-path scoreboard genuinely compares every sampled `rdata`. Because the DUT read
is **combinational** (the generic monitor's split-edge sampling assumes a registered
output), the monitor's `sample_dut_additional` pragma re-samples the inputs at the
same edge as `rdata` — see the comment there. This is a good illustration of the
monitor/predictor pragmas for a 0-latency DUT.

## Layout
- `rtl/regfile.sv` — the register-file DUT (the real one; `gen/regfile.sv` is the
  generated stub, unused by the sim filelist).
- `ral/regfile_ral_pkg.sv` — the external `uvm_reg_block` (reset values + backdoor
  slices match the RTL).
- `regfile.yaml` — config: the `host` agent, `register_model` (package/block/adapter/
  `backdoor_root`), and `csr_tests`.
- `gen/` — generated TB; the user-filled pragmas are:
  - `reg_adapter.svh` `reg2bus`/`bus2reg` — map `uvm_reg_bus_op` ↔ `host_seq_item`.
  - `host_driver.svh` `drive_item_additional` — single write, then sample the read data.
  - `host_monitor.svh` `sample_dut_additional` — re-sample inputs for the combinational read.
  - `regfile_reference_model.svh` `prediction_logic` — the golden regfile model.
- `sim/xrun.f` — Xcelium filelist (`ral/regfile_ral_pkg.sv` compiled before the tb
  package; `rtl/regfile.sv` as the DUT).

## Run it
```sh
cd sim
xrun -f xrun.f +UVM_TESTNAME=rand_test                    # data-path scoreboard (42/42)
xrun -f xrun.f +UVM_TESTNAME=regfile_csr_hw_reset_test
xrun -f xrun.f +UVM_TESTNAME=regfile_csr_bit_bash_test
xrun -f xrun.f +UVM_TESTNAME=regfile_csr_rw_test
```
All four are GREEN on Xcelium (0 UVM_WARNING / 0 UVM_ERROR / 0 UVM_FATAL).

## V2 — register functional coverage from the RAL

`register_model.coverage: true` generates a coverage collector (`regfile_reg_cov`) that installs a
`uvm_reg_cbs` on every field, so each predicted access reports its exact register + read/write kind.
It tracks per-register read/write access — from the RAL's own register list at run time, so it needs
no covergroups in the RAL — plus a per-register covergroup, and reports at end of test, naming any
un-exercised register. It is **access-aware** (a read-only register is never expected to be written,
so it is not reported as a gap), counts RAL-predicted front-door accesses, and measures access — not
field values. It differentiates the CSR tests, which is how you know it has teeth:

```
regfile_csr_rw_test        [REG_COV] register access coverage: 4/4 read, 4/4 written
regfile_csr_bit_bash_test  [REG_COV] register access coverage: 4/4 read, 4/4 written
regfile_csr_hw_reset_test  [REG_COV] register access coverage: 4/4 read, 0/4 written
                           [REG_COV] never written: reg_model.ctrl   (cfg, scratch, status)
```
